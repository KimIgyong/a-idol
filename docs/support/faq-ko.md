## 1. 포토카드 (확률·중복·교환)

### Q1. 포토카드 뽑기 확률 알려주세요

**A.**
모든 포토카드 세트는 **카드별 확률을 공개**하고 있습니다. 상점의 포토
카드 팩 행에서 "📊 확률 공개 (합계 100%)" 버튼을 탭하면 등급별 정렬로
확률이 펼쳐집니다. 예를 들어 HYUN 1st Photocard Set은:

- COMMON (Stage A, Stage B 등) — 각 약 17~23%
- RARE — 각 약 8.5%
- EPIC — 약 3.4%
- LEGENDARY (Debut Night) — 약 0.56%

확률은 서버에서 `dropWeight`를 기반으로 실시간 계산되며 클라이언트가
임의로 변경할 수 없습니다.

**내부 참고.** ADR-016. 공개는 게임산업법 2024 개정(§22 준수 요구)
+ Apple §3.1.1 + Google Play policy 대응. 확률 조정은 관리자 CMS
`/photocards/sets/:id` → 템플릿 dropWeight 수정 → 전체 확률 자동
재계산. 변경 시점은 아직 ledger에 저장 안 함 (T-046 follow-up).

### Q2. 중복 카드 교환하고 싶은데요 / 친구한테 선물할 수 있나요?

**A.**
현재는 포토카드 **교환·선물·환전 기능을 제공하지 않습니다.**

- 게임산업법 §32 ①.7(환전 금지)에 대한 법무 검토가 진행 중이며,
  결과가 나올 때까지 trade/gift/burn/마켓 모두 미구현 상태입니다.
- 중복 카드는 컬렉션에서 `×3` 같은 수량 표시로 확인하실 수 있습니다.
- 법무 검토가 마무리되면 공지사항으로 별도 안내드릴 예정입니다.

**내부 참고.** ADR-018. "현재 법적 검토 중"은 실제 상태 — **absolute
never**로 답변하지 말 것. "기능 추가 일정은 지금 공개할 수 없다" 선.
`Stage A ×9` 같은 중복은 집계 뷰(T-046b)에서 자연스럽게 노출되므로
"저장은 되고 있다" 재확인.

### Q3. 뽑기했는데 LEGENDARY 한 장도 안 나왔어요. 천장(pity) 있나요?

**A.**
MVP에는 천장 시스템이 없습니다. 각 팩은 공개된 확률 그대로 독립적으로
롤되며, 연속 뽑기 시 LEGENDARY 확률이 상승하지 않습니다.

- LEGENDARY 기본 확률은 세트마다 다르지만 통상 1% 미만입니다.
- 5장 팩 기준으로 최소 한 장의 LEGENDARY가 나올 기대 확률은 약
  2.7~4.9% 수준이며, 연속 뽑기에서 모두 빠지는 경우도 수학적으로
  정상입니다.
- 향후 천장 도입 여부는 실제 구매 데이터와 유저 피드백을 기반으로
  검토할 예정입니다.

**내부 참고.** ADR-016 "pity deferred" 결정. 환불 요청 대응 기준은
Q8 참고. 특정 유저의 실제 pull 히스토리는 `user_photocards` 테이블
조회 (`userId` + `obtainedAt` range).

## 2. 투표권

### Q4. 투표권 `5 (3+2)`라고 뜨는데 숫자가 이상해요

**A.**
`5 (3+2)`는 **총 5장 (라운드 전용 3장 + 글로벌 2장)**을 뜻합니다.

- **글로벌 투표권** — 어느 오디션 라운드에서든 사용 가능한 기본 투표권.
- **라운드 전용 투표권** — 특정 라운드 프로모션으로 받은, 그 라운드
  에서만 쓸 수 있는 투표권. 해당 라운드가 종료되면 더 이상 사용
  불가능해집니다.

투표 시 **라운드 전용 투표권부터 자동으로 소진**되며, 다 떨어지면
글로벌 투표권이 사용됩니다. 이렇게 배치한 이유는 유효 기간이 있는
라운드 전용 투표권을 먼저 쓰도록 돕기 위해서입니다.

**내부 참고.** T-062b + ADR-015 활성 계획. Vote UI 뱃지 표시 로직은
`rounds/[id]/vote.tsx`의 `roundTickets + globalTickets`. 실제 버킷 상태는
`round_vote_ticket_balances(userId, roundId)` + `vote_ticket_balances(userId)`
조회. 이중 장부 — ledger 두 개(`round_vote_ticket_ledger`,
`vote_ticket_ledger`)를 시간순으로 union 해서 본다.

---

### Q5. 라운드 끝났는데 그 라운드용 투표권 아직 남아있어요. 환불되나요?

**A.**
라운드가 종료되면 해당 라운드 전용 투표권은 **사용 불가능해지며 환불
대상이 아닙니다**. 구매 시 "이 라운드에서만 유효" 안내가 명시됩니다.

- 글로벌 투표권은 유효 기간 제한이 없어 다른 라운드에서 계속 사용
  가능합니다.
- 라운드 전용 투표권은 프로모션 성격(보통 할인가)으로 판매되며, 만료
  리스크를 반영한 가격입니다.

**내부 참고.** 만료 sweep 로직 없음 — 자동 소각 안 하고, 라운드가
CLOSED가 되면 `CastTicketVoteUseCase`가 `VOTE_ROUND_NOT_ACTIVE`로
거부하기 때문에 de facto 만료. DB의 `round_vote_ticket_balances` 행
은 남아 있지만 사용 불가 — 유저에게 "분실됐다" 대신 **"해당 라운드
한정"**으로 안내할 것. 대량 환불 요구가 들어오면 CS 에스컬레이션.

---

### Q6. NOT_ENOUGH_TICKETS 오류 / HTTP 402 떠요

**A.**
투표권 잔량이 0일 때 표시되는 메시지입니다. 상점에서 투표권 팩을
구매하시면 즉시 적용됩니다.

- 프로필 → 상점 → "투표권" 탭에서 구매 가능.
- 현재 결제는 **DEV_SANDBOX**(테스트 환경)로만 작동합니다. App
  Store / Google Play / 카드 결제는 출시 직후 순차 활성화됩니다.
- **라운드 전용 투표권과 글로벌 투표권 구분**은 Q4 참고.

**내부 참고.** HTTP 402 = PAYMENT_REQUIRED = `NOT_ENOUGH_TICKETS`.
양쪽 버킷(글로벌 + 라운드) 모두 0일 때만 발생. 한 쪽만 0이면 다른
쪽에서 자동 fallback.

## 3. 결제 · 환불

### Q7. 구매했는데 돈이 빠져나갔는지 모르겠어요 / 영수증은 어디에?

**A.**
현재 상점은 **DEV_SANDBOX** 모드로, 실제 결제 금액은 청구되지
**않습니다.** 표시된 가격은 출시 이후 실제 결제에 적용될 금액을 미리
보여드리는 참고값입니다.

- 실결제 시스템(Apple IAP / Google Play Billing / Stripe)은 순차
  도입 예정입니다.
- 활성화 시점은 앱 공지사항과 이메일로 안내드립니다.

**내부 참고.** ADR-015. MVP는 DEV_SANDBOX만 수락, APPLE_IAP /
GOOGLE_IAP / STRIPE는 `PROVIDER_NOT_SUPPORTED` 400 반환 상태.
`purchase_transactions` 테이블에 영수증 기록 (provider=DEV_SANDBOX).
**청소년 결제 한도**는 법무 의견 (`docs/legal/youth-payment-limit-brief-ko.md`)
확정 전까지 공식 안내 금지 — 문의 시 PO 에스컬레이션.

### Q8. 뽑기했는데 원하는 카드가 안 나왔어요. 환불해주세요

**A.**
공개된 확률로 롤된 랜덤 결과는 환불 대상이 아닙니다. 이는 "확률형
아이템" 일반 규정을 따르며, 구매 전 상점 화면에 각 카드별 확률이
공개되어 있습니다.

다음 경우에는 **예외적으로 검토**합니다:

- 시스템 오류로 구매가 이중 처리된 경우 (요청 ID를 함께 알려주세요)
- 결제는 완료되었으나 카드가 인벤토리에 들어오지 않은 경우
- 상점의 공개 확률과 실제 분포가 명백히 다른 것으로 확인되는 경우

환불을 요청하실 때는 앱의 **에러 화면에 표시되는 요청 ID**
(`요청 ID: abc12345…xyz9876`)를 메시지에 붙여주시면 조사 속도가
빨라집니다.

**내부 참고.** ADR-016. "확률 공개 했으니 운 나쁜 건 환불 대상 아님"
은 표준 대응. 이중 과금/미지급은 `purchase_transactions`에 동일
`providerTxId` 중복 체크 — `(provider, providerTxId)` UNIQUE 제약으로
이미 차단되지만, 0이벤트 발생 시 R-03 확인 후 수동 보정.

## 4. 채팅

### Q9. 채팅 하루 메시지 한도는 언제 초기화되나요?

**A.**
**매일 한국 시간(KST) 자정 0시**에 초기화됩니다. 자정 기준 전후로
메시지를 보내시면 날짜가 바뀐 시점에 잔량이 자동 리셋됩니다.

- 무료 한도를 다 쓴 후에는 **채팅 쿠폰**을 사용해 추가로 메시지를
  보내실 수 있습니다.
- 쿠폰은 유효 기간이 없으며 일일 한도와 별개로 누적·사용됩니다.

**내부 참고.** KST = UTC+9. 초기화는 두 경로로 보장 — lazy reset
(메시지 보낼 때 `ChatBillingRepository.consumeOne`이 KST 날짜 체크
후 리셋) + BullMQ 크론 백스톱(`0 0 * * *` KST). 쿠폰은
`chat_coupon_wallets` + `chat_coupon_ledger`에서 조회.

---

### Q10. 채팅방에서 아이돌이 같은 말만 반복해요

**A.**
현재 MVP의 아이돌 채팅은 **키워드 기반 룰 엔진**으로 작동합니다.
자연스러운 AI 응답은 베타 이후 제공될 예정이며, 지금은 일부 키워드에
대해 사전 정의된 답변이 나옵니다.

- 오작동이 반복되면 요청 ID와 함께 신고해주세요.
- 정식 AI 채팅(persona-based)은 별도 출시 안내가 있을 때까지 기다려
  주세요.

**내부 참고.** ADR-013. `RuleBasedReplyEngine` 상태. T-031/T-032 확장
시 LLM 연결 예정이며 현재 로드맵 외. "AI 같다"고 답변하지 말 것.

## 5. 계정 · 오류 메시지

### Q11. 앱에서 에러가 떴는데 "요청 ID"가 뭔가요?

**A.**
"요청 ID"는 오류가 발생한 해당 요청의 고유 식별자로, 지원팀이 로그를
빠르게 찾는 데 사용됩니다.

문의 시 다음 중 하나를 함께 보내주시면 좋습니다:

- 에러 화면의 **요청 ID** 박스를 탭하면 클립보드에 복사됩니다.
- 또는 스크린샷에 표시된 짧은 형태(`abc12345…xyz9876`)를 그대로
  입력해주세요.

요청 ID 없이도 문의 접수는 가능하지만, 있으면 조사 소요가 평균
30분 → 수 분 수준으로 단축됩니다.

**내부 참고.** ADR-017. 백엔드 로그에서 `reqId=<UUID>`로 grep —
nestjs-pino가 모든 로그 라인에 첨부. Sentry는 환경 준비 후 연동
예정이며 같은 필드에 들어갈 것. 4xx 비즈니스 룰 에러에는 id가
노출되지 않으니 "요청 ID가 안 보여요" 문의 시 **"어떤 화면에서?"**
부터 확인.

---

### Q12. 회원가입이 안 되고 "만 14세 미만" 에러가 떠요

**A.**
개인정보보호법상 **만 14세 미만** 이용자는 법정대리인 동의 절차가
별도로 필요한데, 현재 당사는 해당 동의 시스템을 아직 제공하지 않고
있어 만 14세 미만 가입을 일괄 차단하고 있습니다.

- 생년월일 입력 시 연령이 확인되며, 14세 이상이시면 정상 가입
  가능합니다.
- 연령 계산은 가입 시점 기준 만 나이(생일 기준)입니다.
- 법정대리인 동의 기반 가입 절차는 향후 도입 예정이며 공지로
  안내드립니다.

**내부 참고.** `ErrorCodes.UNDER_AGE` → HTTP 422. `User.assertMinimumAge`
(domain 로직)가 signup 입장 전에 검증. 법무 브리프 (`docs/legal/
youth-payment-limit-brief-ko.md`) 답변에 따라 청소년 결제 한도 + 법정
대리인 동의 UI가 함께 설계될 예정.

---

### Q13. 로그인이 안 돼요 / 비밀번호를 잊어버렸어요

**A.**
이메일과 비밀번호를 다시 확인해주세요. 올바른 조합인데도 실패하신다면
다음 경우일 수 있습니다:

1. **소셜(카카오/애플) 계정으로 가입**하신 경우 — 이메일·비밀번호
   로그인 대신 원래 가입하신 소셜 버튼으로 로그인해주세요.
2. **다른 이메일로 가입**하셨을 가능성 — 다른 주소 확인 바람.

비밀번호 재설정 기능은 **현재 MVP 범위 외**이며 정식 출시 시점에
이메일 인증 기반으로 제공될 예정입니다. 그 전에 비밀번호를 잊어버리신
경우, 지원팀에 문의하시면 계정 확인 후 재설정을 도와드립니다.

**내부 참고.** `INVALID_CREDENTIAL` (HTTP 401)은 (a) 존재하지 않는
이메일 (b) 비밀번호 틀림 (c) 소셜 가입 계정에 비밀번호 로그인 시도
— **세 경우 모두 동일 에러 코드**로 반환 (계정 존재 여부 은닉,
타이밍 공격 방어). 실제 원인 확인은 백엔드 `reqId` + `user` 쿼리로만
가능 — 지원 시 요청 ID 필수.

### Q14. 앱을 쓰다가 갑자기 로그아웃됐어요 / 자주 다시 로그인해야 해요

**A.**
안전 조치로 **의심 활동이 감지되면 세션이 자동 해제**됩니다. 가장 흔한
사유는:

1. **여러 기기에서 동시 사용** — 리프레시 토큰이 두 기기에서 겹쳐
   사용되면 보안상 해당 세션 전체를 무효화합니다. 주 사용 기기에서
   다시 로그인하시면 됩니다.
2. **앱이 오랫동안 백그라운드** — 리프레시 토큰 유효기간(14일)이 지나
   재로그인 필요.
3. **비정상 네트워크 상황** — 앱이 재시작될 때 과거 토큰으로 재시도하면
   만료 처리됩니다.

반복적으로 자주 로그아웃된다면 요청 ID와 함께 신고 부탁드립니다.

**내부 참고.** `RefreshTokenUseCase` (TC-REFRESH-005) — 해시 mismatch
감지 시 방어적으로 `sessions.revoke(sid)` 호출 → 해당 디바이스 세션
통째 무효. 유저가 "한쪽 기기에서 로그아웃됨"을 경험하는 정상 동작.
세션 재사용 공격 대응이라 의도된 방어선 — 사용자에게는 **"재로그인
해달라"**로만 안내하고 내부 혼동 방지.

### Q15. 채팅하려면 꼭 팬클럽에 먼저 가입해야 하나요?

**A.**
네. **아이돌 채팅은 해당 팬클럽 멤버에게만 열려 있습니다**.

1. 아이돌 상세 화면 → "팬클럽 가입" 버튼 (무료).
2. 가입 즉시 채팅 가능.

팬클럽은 현재 모두 무료로, 언제든 탈퇴 가능합니다. 유료 팬클럽 옵션은
Q16 참고.

**내부 참고.** `CHAT_GATE_NOT_MEMBER` (HTTP 403) — `OpenRoomUseCase`가
`ChatMembershipChecker.isActiveMember` 통과 못하면 거부. ADR-013 게이트.
유저 의도(팬 라이언스 확보 + 모더레이션 표면 축소)는 제품 결정이지
버그 아님.

### Q16. 팬클럽 유료 가입 언제 열려요?

**A.**
유료 팬클럽 구독은 **MVP 이후 출시 예정**입니다. 현재는 모든 팬클럽이
무료 가입·탈퇴만 지원하며, 결제 시스템(IAP / Stripe) 도입과 함께
단계적으로 활성화됩니다.

- 출시 일정은 앱 내 공지로 별도 안내드립니다.
- 가입한 팬클럽은 계속 유지되며, 유료 전환 시 기존 무료 멤버에게
  혜택이나 전환 옵션이 제공될 수 있습니다(세부 정책 추후 공지).

**내부 참고.** ADR-012. `FanClubSubscriptionFulfiller`는 ADR-015 활성
계획에 명시만 되어 있고 구현 미시작. 현재 `FanClub.type`는 FREE 전용
수락, PAID는 API가 `PAID_FAN_CLUB_NOT_SUPPORTED` 반환.

## 6. 에스컬레이션 가이드

법적/규제 민감한 질문(게임산업법, 환전, 확률 조작 의혹 등)은 **즉시
PO에게 에스컬레이션**. 즉흥 답변 금지.

지원팀이 단독으로 답변해도 되는 범위:

- ✅ 기능 사용법 (어디서 누르면 되나요)
- ✅ 현재 기능 상태 (언제 열려요 → "출시 후 안내")
- ✅ 오류 원인 확인 + 요청 ID 안내
- ✅ Q2 · Q5 · Q7 · Q8 · Q12 · Q15 · Q16 같은 정책·운영 질문 중 이 문서에 등재된 범위

에스컬레이션 필수:

- ❌ 환불 특정 사례 개별 판단 (Q8의 예외 항목)
- ❌ 확률 조작 · 결과 조작 주장
- ❌ 개인정보 · 삭제 요청 (GDPR / 개인정보보호법 건)
- ❌ 언론/미디어 문의
- ❌ 법률대리인 명의 통지

모든 에스컬레이션은 요청 ID 포함해서 `#support-escalation` 채널에
전달.
